JSON-LD
Specification

v1.0 Draft May 2026 Schema: tracestandard.org/schema/v1

The JSON-LD format embeds a TRACE assessment as structured linked data in the <head> of a web page or as a field in an API response. It is indexed by search engines, readable without tooling, and requires no dependencies beyond a script tag.

Context & Type

Declaring a TRACE Assessment

All TRACE JSON-LD objects must declare the schema context and the TRACEAssessment type. The context URL is the canonical location of the schema definition.

Minimal valid declaration 
"@context": "https://tracestandard.org/schema/v1",
"@type":    "TRACEAssessment"

The schema does not currently extend a Schema.org type but is designed to coexist with schema:CreativeWork in the same JSON-LD block. If embedding alongside Schema.org, declare both contexts in an array:

Co-embedding with Schema.org
"@context": [
  "https://schema.org",
  "https://tracestandard.org/schema/v1"
],
"@type": ["CreativeWork", "TRACEAssessment"]
Schema Reference

Property Definitions

Properties marked required must be present in all valid assessments. Properties marked Track B only apply only when track is "B".

Property Type Status Description
@context URI string required Must be "https://tracestandard.org/schema/v1" or an array including it.
@type string required Must be "TRACEAssessment" or an array including it.
contentTitle string required Title or descriptive name of the content being assessed.
contentURL URI optional Canonical URL of the content. Recommended when the assessment is embedded in a different page from the content itself.
contentType enum optional One of: podcast, article, video, image, audio, document, series, other.
producer string required Name of the individual, outlet, or organization that produced the content.
producerURL URI optional Home page or profile URL of the producer.
communityRepresented string required Free-text description of the community or communities the content is about, from, or produced on behalf of.
aiGenerated boolean optional Whether AI tools played a role in generating the content. Omit only if genuinely unknown; prefer false over omission when no AI was used.
aiTools string[] optional List of AI tools or systems used. Relevant only when aiGenerated is true. Example: ["Claude", "ElevenLabs"].
aiRole enum optional One of: primary (AI wrote the content), partial (AI wrote portions), assisted (AI helped with research or editing), none.
assessmentDate date (ISO 8601) required Date the assessment was completed. Format: YYYY-MM-DD.
assessedBy string optional Who conducted the assessment. Use "self" for self-assessment, "third-party" for independent review, or a name/organization.
nextReviewDate date (ISO 8601) optional Recommended date for reassessment. Useful for ongoing projects where conditions may change.
track enum required One of: "A" (Community-Insider) or "B" (External/Institutional).
disqualified boolean required Whether a disqualifier (DQ1 or DQ2) was triggered. If true, score and verdict fields should be omitted.
disqualifierTriggered enum optional Which disqualifier applied. One of: "DQ1" (authorship misrepresentation), "DQ2" (unconsented attribution). Omit when disqualified is false.
scores object optional Category-level scores. See Scores Object below. Omit when disqualified is true.
totalScore integer required Total points awarded. Must be ≥ 0 and ≤ maxScore. Omit when disqualified is true.
maxScore integer required Maximum possible score for this track. Must be 20 for Track A or 35 for Track B. Omit when disqualified is true.
verdict enum required See Verdict Values. Omit when disqualified is true; use "Disqualified" if you prefer to include it.
cardURL URI optional URL of the shareable TRACE scorecard generated by the assessment tool.
frameworkVersion string optional Version of the TRACE framework used for assessment. Current value: "1.0".
Scores Object

Category Score Structure

The scores property is an object where each key is a category name and each value is a ScoreObject with two integer fields: score (points awarded) and max (maximum for that category).

ScoreObject structure
{ "score": integer, "max": integer }
Category keyTrack A maxTrack B maxNotes
transparency47Track B max is higher due to the Funding & Motivation Disclosure sub-criterion.
sourceIntegrity34Track B includes a Compensation criterion not scored at Track A scale.
displacement25Track B adds an Investment in Community Creator Capacity criterion.
consentAttribution44Equal across tracks. May be reduced if N/A applied (document in notes).
benefitFlow56Track B's Archive Ownership sub-criterion is worth 2 points vs Track A's 1.
accountability24Track B adds an Editorial Authority criterion.
governanceDurabilityN/A5Track B only Must be omitted for Track A assessments.
N/A handling: When consentAttribution sub-criteria are scored N/A (no living people represented), include the score as if N/A = 2 points, but add a consentAttributionNote string property to the scores object documenting the rationale. Example: "consentAttributionNote": "No living individuals identified or represented in this content."

Allowed Verdict Strings

The verdict field must contain one of the strings below, depending on track and score. Use exact casing as shown.

Embedding in HTML

Adding TRACE Metadata to a Web Page

Place the JSON-LD object in a <script type="application/ld+json"> tag inside the page's <head>. Multiple JSON-LD blocks are permitted on the same page.

HTML embedding pattern
<head>
  <meta charset="UTF-8">
  <title>Episode 12: Water Rights on the Chilcotin</title>

  <!-- TRACE Assessment -->
  <script type="application/ld+json">
  {
    "@context": "https://tracestandard.org/schema/v1",
    "@type": "TRACEAssessment",
    "contentTitle": "Episode 12: Water Rights on the Chilcotin",
    "producer": "Cariboo Signals",
    "communityRepresented": "Cariboo region, BC",
    "track": "A",
    "disqualified": false,
    "totalScore": 15,
    "maxScore": 20,
    "verdict": "Sound",
    "assessmentDate": "2026-05-15"
  }
  </script>
</head>
CMS note: Many CMS platforms (WordPress, Ghost, Webflow) allow injecting HTML into the <head> via a "custom code" or "header scripts" field. The JSON-LD block can be added there without plugin support. For per-post assessments, use a custom field and template the values in.
Validation

Checking Your Implementation

A TRACE JSON-LD validator is forthcoming at tracestandard.org/validate. In the meantime, use these tools to catch structural issues:

ToolWhat it checksURL
JSON-LD Playground Parses and validates JSON-LD syntax; expands the context graph json-ld.org/playground
Google Rich Results Test Confirms structured data is detectable from a live URL or code snippet search.google.com/test/rich-results
Schema Markup Validator Schema.org validator; catches malformed JSON and missing required fields against known types validator.schema.org
Required field check: A valid non-disqualified TRACE JSON-LD object must include: @context, @type, contentTitle, producer, communityRepresented, assessmentDate, track, disqualified, totalScore, maxScore, and verdict. Disqualified assessments omit the last three.
